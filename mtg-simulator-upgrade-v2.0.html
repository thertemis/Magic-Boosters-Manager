README.md

MTG Pack Simulator

A Magic: The Gathering booster pack simulator with economy, marketplace, scheduled grants, custom booster DSL, Scryfall syntax support, decklist checking, a full REST API, and EN/FR language support.

Features

  • Booster Pack Opening — Animated 3D card reveals with confetti for rares. Supports classic 15-card, modern 14-card play booster, and collector booster formats (auto-detected by set release date)
  • Card Pool Editor — Enable/disable individual cards from packs; filter with full Scryfall syntax; image modal on click
  • Custom Booster DSL — Define custom pack compositions with rarity/Scryfall filters and probability slots
  • Economy & Marketplace — Currency system, pack store, user-to-user card trading, sell-to-store at Scryfall prices
  • Automatic Schedules — Auto-grant boosters to players daily at a configured UTC hour
  • Decklist Check — Paste any MTG decklist and see which cards you own vs. which are missing
  • REST API — Full API v1 with API key authentication for external integrations
  • Language Support — English and French UI (selectable per user)
  • Backup & Restore — Full JSON export/import of all data

    • Stack

  • Frontend: React 18 + TypeScript, Vite, Wouter, TanStack Query, Shadcn/ui, Framer Motion, Tailwind CSS
  • Backend: Express.js + TypeScript (tsx dev, esbuild prod)
  • Database: PostgreSQL via Drizzle ORM
  • Auth: Passport.js (local strategy) + express-session

    Docker Upgrade Guide — Zero Data Loss

    This guide explains how to safely upgrade your running MTG Pack Simulator instance to the latest version without losing any data.

    Overview

    The application uses Drizzle ORM with schema push — no migration files are needed. When you run npm run db:push, it:

  • Creates any new tables
  • Adds any new columns with default values
  • Never deletes existing data
  • Skips columns that already exist

    • This means upgrading the database schema is always safe to run on a live database.

    Step 1: Back Up Your Database (Strongly Recommended)

    Option A — Use the in-app backup (Admin → Backup → Export All Data) Download the JSON file and keep it somewhere safe.

    Option B — PostgreSQL dump 

    # Find your postgres container name
docker ps
    Dump the database
    
docker exec <postgres-container-name> \
  pg_dump -U <db_user> <db_name> > backup_$(date +%Y%m%d_%H%M%S).sql
    Example:
    
docker exec mtg-db pg_dump -U mtg mtgsim > backup_20260321.sql

    Step 2: Pull the New Image

    If building from source (Replit / GitHub): 

    git pull origin main
docker build -t mtg-simulator:latest .

    If using a container registry: 

    docker pull yourregistry/mtg-simulator:latest

    Step 3: Stop and Remove the Old Container

    The application data lives in PostgreSQL, not in the container, so removing it is safe.

    # Check running containers
docker ps
    Stop and remove old container
    
docker stop mtg-simulator
docker rm mtg-simulator

    Step 4: Start the New Container

    docker run -d \
  --name mtg-simulator \
  --env DATABASE_URL=postgresql://mtg:changeme@mtg-db:5432/mtgsim \
  --env SESSION_SECRET=your-secret-here \
  -p 5000:5000 \
  --restart unless-stopped \
  mtg-simulator:latest

    The schema migration happens automatically when the app starts (via npm run db:push in the startup sequence).

    If you need to run it manually: 

    docker exec mtg-simulator npm run db:push

    Step 5: Verify

    # Check logs for successful startup
docker logs mtg-simulator --tail 50
    Expected output:
    

    [express] serving on port 5000
    

    (no errors about missing tables or columns)

    # Pull new image
docker compose pull
    Restart with new image (data volume is preserved)
    
docker compose up -d
    Run schema sync if needed
    
docker compose exec app npm run db:push

    What's New in v2.0 — New Database Tables

    The following tables are added automatically by db:push. They are new and empty by default — no existing data is affected:

    New Table | Purpose ---|--- economy_settings | Global economy configuration user_balances | Per-player currency balances currency_transactions | Full transaction history market_pack_listings | Admin-created pack store items market_card_listings | Player-to-player card listings booster_schedules | Daily auto-grant schedules api_keys | REST API authentication keys

    The following columns are added to existing tables:

    Table | New Column | Default ---|---|--- users | preferred_language | 'en'

    Restoring From Backup (If Needed)

    If something goes wrong, restore from your backup:

    From in-app JSON backup: 1. Start the app on a clean database 2. Go to Admin → Backup → Import All Data 3. Upload your JSON file

    From SQL dump: 

    # Copy dump into postgres container
docker cp backup_20260321.sql mtg-db:/tmp/
    Restore
    
docker exec mtg-db psql -U mtg mtgsim < /tmp/backup_20260321.sql

    Important Notes

  • Never change primary key column types in the schema — this generates destructive SQL
  • db:push is safe to run multiple times — it is idempotent
  • If db:push prompts about column drops/renames interactively, choose to keep existing columns (select "No")

    GitHub Update Guide

    How to sync your MTG Pack Simulator to GitHub and keep it up to date.

    First Time: Connect to GitHub

    From Replit

    1. Open your Replit project 2. In the shell, initialize git and connect to your repo:

    git init
git remote add origin https://github.com/yourusername/mtg-simulator.git
git branch -M main

    Recommended .gitignore

    Create a .gitignore file at the project root:

    node_modules/
dist/
.env
.env.local
*.log
.replit
replit.nix
attached_assets/
upgrade-archive/

    Pushing Your Changes to GitHub

    1. Check what has changed

    git status
git log --oneline -5

    2. Stage your changes

    # Stage everything
git add -A
    Or stage specific files/folders
    
git add shared/ server/ client/src/ package.json

    3. Commit

    Use clear, descriptive commit messages:

    git commit -m "feat: add economy, marketplace, decklist check, API keys"

    Commit message prefixes:

  • feat: — new feature
  • fix: — bug fix
  • chore: — dependencies, config, cleanup
  • docs: — documentation only
  • db: — database schema changes (always note these!)

    • 4. Push to GitHub

    git push origin main

    If the remote has commits you don't have locally: 

    git pull --rebase origin main
git push origin main

    Pulling Updates Into Your Replit

    If someone else (or another device) pushed changes to GitHub:

    # Pull latest changes
git pull origin main
    Install any newly added npm packages
    
npm install
    Apply any new database columns/tables (safe, no data loss)
    
npm run db:push

    After a Database Schema Change

    Whenever a commit includes changes to shared/schema.ts, run on all environments:

    npm run db:push

    This is safe to run on a live database — it only adds, never removes data.

    Tagging a Release

    When you reach a stable version, tag it:

    git tag -a v2.0.0 -m "Economy, marketplace, API, decklist check, i18n"
git push origin v2.0.0

    Then on GitHub, go to Releases → Create a release from the tag.

    Keeping Secrets Out of Git

    Never commit .env or any file containing:

  • DATABASE_URL
  • SESSION_SECRET
  • API keys

    • Use Replit's Secrets panel for development, and Docker environment variables for production.
